How Sprint Planner Works

Import your JIRA backlog, configure your team, and get an auto-calculated sprint schedule with dependency-aware timelines, resource allocation, and sprint-wise breakdowns.

1

Export from JIRA

Go to your JIRA board or backlog, use Filters → Export, and download as CSV or Excel (.xlsx). The export must include these columns:

Issue TypeEpic, Story, Sub-task, Bug, Task
Issue Keye.g. PROJ-123
SummaryTask title/description
AssigneeEngineer name
StatusTo Do, In Progress, Done
Original Estimatee.g. 8h, 1d 4h, 3600
Fix Version/sGroups stories by release
Parent KeyLinks sub-tasks → stories → epics
SprintJIRA sprint name
The tool auto-detects column name variations (e.g. "Issue key" vs "Issue Key"). Items with status Done/Closed/Resolved are automatically excluded from scheduling.
2

Configure Your Team

After upload, the tool auto-detects all assignees. For each engineer, configure:

Role Assign BE FE or Test to each engineer. This determines scheduling order: BE → FE → Test (dependency chain).
Hours/Day Productive hours per day. Used to convert hour estimates into working days. A task estimated at 16h with 8h/day = 2 working days.
Leave Dates Add specific dates when each engineer is unavailable. The scheduler skips these days when assigning work.
3

Set Sprint Dates & Holidays

Choose between two modes:

New Sprint Set Sprint Start, Testing Start, and Sprint End dates. Dev tasks begin from Sprint Start; test tasks begin from Testing Start date.
Ongoing Sprint Set Dev Start, Testing Start, and Sprint End. Use this when a sprint is already in progress and you want to re-plan remaining work.

Add Holidays (team-wide non-working days) and define additional Sprint Periods to see multi-sprint breakdowns.

4

Calculate & Analyze

Click Calculate (or press Ctrl+Enter) to generate the schedule. The engine:

aSchedules all BE tasks first, starting from the sprint start date, respecting each engineer's availability.
bSchedules FE tasks next — they can start in parallel but their end date must be ≥ 1 day after related BE work completes.
cSchedules Test tasks last — they only begin after ALL dev (BE+FE) subtasks of the same story are complete, and never before the Testing Start date.
dSkips weekends, holidays, and per-engineer leave dates automatically.
eAuto-creates additional sprint periods if tasks overflow the defined sprints.
5

Review the Output Tabs

Task TreeHierarchical view: Epic → Fix Version → Story → Sub-task. Toggle include/exclude checkboxes. Dates appear after calculation.
High-Level TimelineWeekly Gantt chart showing Dev, BE, FE, and Test phase bars, per-engineer schedules, and sprint period overlays. Includes a "Today" marker.
TimelineDay-by-day detailed Gantt chart — every task bar per engineer with exact start/end dates. Weekends highlighted.
Sprint BreakdownPer-sprint story status: Completed, Dev Done (testing carries forward), or Carry Forward. Exportable to Excel.
Resource BreakdownPer-engineer, per-sprint hours allocation table with workload imbalance suggestions.
SummaryEngineer workload overview and story schedule summary tables.

Quick Tips

Exclude items — Uncheck epics, stories, or individual sub-tasks in the Task Tree to remove them from the schedule without deleting data.
Zero estimates — Items with 0h estimate are highlighted in red. They still get scheduled as 1-day tasks. Add estimates in JIRA for accuracy.
Auto-saved config — Engineer roles, hours/day, leave dates, sprint dates, and holidays are saved in your browser. They persist across page reloads.
Export options — Use "Export" for the full plan (all tasks + dates + engineers). Use "Export Sprint Breakdown" in the Sprint Breakdown tab for a sprint-wise story summary.
